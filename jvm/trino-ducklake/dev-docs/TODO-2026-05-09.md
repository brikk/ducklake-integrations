## #1 — Puffin deletion-vector detection guard (read side) — DONE

**Why this was #1**: the only open item in either doc that *silently returned
wrong results today* under a configuration a DuckDB writer might enable
(`write_deletion_vectors=true`). The connector read only `parquet` delete files;
encountering a `puffin` delete file meant rows that should be deleted were
returned to the user. Ship-blocker for any catalog shared with DuckDB writers.

**Landed**:
- `DucklakeDataFile` now carries `Optional<String> deleteFileFormat`, populated
  from `ducklake_delete_file.format` in `JdbcDucklakeCatalog.getDataFiles`.
- `DucklakeSplitManager.validateDeleteFileFormats` throws
  `TrinoException(NOT_SUPPORTED, ...)` for any non-`parquet` delete-file format,
  naming the schema.table, the offending format, and DuckDB's
  `write_deletion_vectors` setting.
- `TestDucklakePuffinDeleteFileGuard` injects a `format='puffin'` catalog row
  (metadata-only — guard runs before any IO, so no real puffin file is needed)
  and asserts the throw.

**Long-term reader** (not in this task; remains open in TODO-READ-MODE):
implement Roaring-bitmap puffin reads on the Trino side using Iceberg's
`iceberg-core` puffin library + `org.roaringbitmap:RoaringBitmap`. Lift the
guard and add a round-trip test with a real DuckDB-written puffin file at
that point.

**Pointer**: source item in [TODO-READ-MODE.md § Puffin Deletion Vector Reads](TODO-READ-MODE.md#puffin-deletion-vector-reads)
(short-term entry now checked off; long-term reader remains).
Reuse analysis: [DUCKLAKE_1_0_IMPACT.md § Existing Puffin Support in Trino](DUCKLAKE_1_0_IMPACT.md#existing-puffin-support-in-trino-reuse-analysis).

## #2 — `uuid` scalar + `list<uuid>` inlined reads (read side) — DONE

**Why this was #2**: concrete cross-engine read failure (DuckDB writes a UUID,
Trino reads inlined → threw `"Expected entry size to be exactly 16"`). UUIDs
are a common primary-key choice, so this was not a corner case.

**Landed**:
- `DucklakeInlinedValueConverter.toUuidSlice` parses the 36-char text via
  `java.util.UUID.fromString` and packs through `UuidType.javaUuidToTrinoUuid`.
  The list path inherits the fix because the list parser delegates per-element
  through `convertJdbcValue`.
- `@Disabled` lifted from `testDuckdbListUuidReadsInTrino`; both inlined and
  Parquet halves now pass (also validates Trino's Parquet array reader against
  DuckDB's UUID layout).
- New `testDuckdbInlinedUuidReadsInTrino` pins scalar-UUID inlined reads
  end-to-end alongside the existing inlined-blob/varchar tests.

**Bonus deferred**: `list<blob>` inlined reads still fail (DuckDB's `\xNN` hex
escapes are not decoded). Test stays `@Disabled` with the same message; tracked
as a still-open bullet in `TODO-READ-MODE.md § Inlined-Read Type Gaps`.

**Pointer**: [TODO-READ-MODE.md § Inlined-Read Type Gaps](TODO-READ-MODE.md#inlined-read-type-gaps)
(uuid bullet now checked off; `list<blob>` bullet remains).

## #3 — Logical conflict checking in commit flow (write side) — DONE (steps 1–4)

**Landed (2026-05-10)**:
- Concurrency test harness `ConcurrentWriterHarness` extracted from
  `TestJdbcDucklakeCatalogConcurrentCommit`; the existing test refactored
  onto it; pinning tests `TestConcurrentInsertSameTable` and
  `TestConcurrentInsertDifferentTables` added.
- Typed `WriteChange` channel: sealed interface with `CreatedSchema /
  DroppedSchema / CreatedTable / DroppedTable / AlteredTable /
  InsertedIntoTable(referencedColumnIds) / DeletedFromTable(referencedDataFileIds) /
  CreatedView / DroppedView / AlteredView`. `DucklakeWriteTransaction`
  now records typed changes via `recordChange(WriteChange)`; the old
  string-based `addChange` API is gone. The
  `ducklake_snapshot_changes.changes_made` serializer
  (`WriteChange.formatChangesMade`) renders them per spec — quoting and
  joining live in one place.
- `LogicalConflictCheck` invoked in `attemptWriteTransaction` between
  the lineage check and the snapshot-row insert. For each typed change,
  validates the referenced entity is still active at
  `tx.getCurrentSnapshotId()`. Throws non-retryable
  `LogicalConflictException`; `WriteTransactionRetry` now consults
  `TransactionConflictException.retryable()` and short-circuits on
  `false` so non-retryable conflicts don't burn the retry budget.
- Acceptance tests: `TestConcurrentInsertVsDropColumn` (gold —
  asserts the dropped column_id is named in the error and `retryable()`
  is `false`), `TestConcurrentInsertVsDropTable`,
  `TestConcurrentAlteredTableVsAlteredTable`.

**Step 4 (port upstream's change-vs-change matrix) — landed
2026-05-10.** New files: `InterveningChanges` (parser + per-upstream
aggregator), `ConflictMatrix` (direct port of
`ducklake_transaction.cpp:1184–1314`); wiring in
`JdbcDucklakeCatalog.attemptWriteTransaction`'s `runConflictMatrix`
(matrix + delete-vs-delete file-overlap query mirroring upstream's
`GetFilesDeletedOrDroppedAfterSnapshot` path at `:1259–1283`). The
matrix runs only when intervening commits exist (matches upstream's
`i > 0` gate). New tests:
`TestConcurrentCreateSchemaSameName`,
`TestConcurrentCreateTableSameName`,
`TestConcurrentCreateTableInDroppedSchema`,
`TestConcurrentDeleteVsDelete`,
`TestInterveningChangesParser`. The behavior pinned in
`TestConcurrentAlteredTableVsAlteredTable` flipped to upstream's
`altered_table × altered_table` conflict policy (`:1307–1310`); test
updated to assert non-retryable conflict.

`WriteChange.{CreatedTable, CreatedView, DroppedSchema}` gained extra
fields (schemaId / schemaName) for the matrix's cross-checks; not
serialized to changes_made.

Conflict-detection now matches upstream's commit-time semantics. The
catalog is at parity for the cases upstream covers, plus the
state-based `LogicalConflictCheck` from Step 3 (which is strictly
stronger on per-call args — upstream doesn't validate
fragment-referenced column / file IDs).

---

**Why this was #3**: the next tier of correctness work, this time on writes.
The catalog already retries on any intervening commit (strict lineage check),
but the retry's action re-runs with **stale per-call arguments** — `tableId`,
column-stats `column_id`s, delete-target `data_file_id`s — which can reference
entities the winner committed away during the original window. The lineage
looks fine; the *meaning* of the change is broken. Concrete failure mode:
T2's `commitInsert` retry inserts `ducklake_file_column_stats` rows pointing
at a column T1 just dropped.

**Engine-agnostic placement**: all changes land in `jvm/ducklake-catalog`.
The Trino plugin only invokes catalog methods (`commitInsert`, `commitDelete`,
`commitMerge`, DDL methods) and translates the catalog's exceptions via
`translateCatalogExceptions` in `DucklakeMetadata.java`; that wrapper already
surfaces `TransactionConflictException` to Trino unchanged. Future engines
(Doris, Spark, etc.) reuse the matrix work as-is — only their own exception
translation needs to be wired up.

**Strict lineage check stays for now**: relaxing it is non-trivial (requires
solving `nextCatalogId` / `nextFileId` allocation across intervening commits)
and is tracked separately as Step 4 in TODO-WRITE-MODE. The acceptance
scenario is fully exercised with strict lineage retained — the logical check
fires on the *retry attempt* whose action re-runs with stale fragment
metadata.

**Plan** (see [TODO-WRITE-MODE.md § Logical Conflict Checking + Concurrency Test Coverage](TODO-WRITE-MODE.md#logical-conflict-checking--concurrency-test-coverage)
for full per-step detail; each step is one branch/PR):

1. **Concurrency test harness + pinning tests** — extract the latch-pause
   pattern from `TestJdbcDucklakeCatalogConcurrentCommit` into a reusable
   fixture; add same-table and cross-table pinning tests. Must pass against
   unmodified production code.
2. **Structured `WriteChange` channel** in `DucklakeWriteTransaction` —
   parallel to today's `addChange(String)` but typed (sealed interface with
   per-op variants carrying target IDs and referenced column/file IDs).
   Mechanical conversion at every catalog write call site.
3. **`LogicalConflictCheck` + acceptance test** — invoked between
   `action.execute(tx)` and `insertSnapshotRow` in `attemptWriteTransaction`
   (`JdbcDucklakeCatalog.java:938`). Validates referenced entities are still
   active at the current snapshot. Acceptance: T1 `DROP COLUMN c` racing
   with T2 `INSERT` against column c → T2 retry fails with a clear
   logical-conflict error naming column c specifically. **Decide explicitly
   per conflict type whether it's retryable** — stale-column INSERT is not
   (re-running fails identically); add a non-retryable subclass or flag if
   needed.
4. **Port upstream's change-vs-change matrix** — landed 2026-05-10.
   Catches dueling-name commits the state-based check in Step 3 missed
   (upstream's DDL omits PKs on snapshot-versioned tables, so the
   matrix is the only safety net). Direct port of
   `ducklake_transaction.cpp:1184–1314`.

**Pointer**: [TODO-WRITE-MODE.md § Logical Conflict Checking + Concurrency Test Coverage](TODO-WRITE-MODE.md#logical-conflict-checking--concurrency-test-coverage).

## Considered and deferred

These came up in triage and were deliberately *not* in the top 3:

- **`list<blob>` inlined reads** — correctness bug like UUID, but blobs in
  lists are rarer and the failure mode is silent garbage rather than a throw.
  Ride-along with #2 if cheap; otherwise queue.
- **Bucket partitioning, sorted writes, schema-evolution gaps (`SET TYPE`,
  nested struct field manipulation), commit context (`set_commit_message`),
  M8 maintenance procedures** — real feature backlog, not correctness bugs.
  Pick up after #1–#3.
- **Variant / Geometry / virtual columns** — degraded-but-honest is the
  documented stance. Ship later or never.
- **Sorted-table awareness on read, change feed (R6), cross-backend view
  tests (R7), `.slt` corpus adoption, view dialect transpilation** — research
  / optimization items. Not blocking anything correctness-relevant.
