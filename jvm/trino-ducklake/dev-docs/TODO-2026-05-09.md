# Top 3 Priorities — 2026-05-09

Triaged from [TODO-READ-MODE.md](TODO-READ-MODE.md) and [TODO-WRITE-MODE.md](TODO-WRITE-MODE.md)
by correctness risk first, then effort-to-value. Pick up in order.

## #1 — Puffin deletion-vector detection guard (read side)

**Why this is #1**: the only open item in either doc that *silently returns wrong
results today* under a configuration a DuckDB writer might enable
(`write_deletion_vectors=true`). The connector reads only `parquet` delete files;
encountering a `puffin` delete file means rows that should be deleted are returned
to the user. Arguably a ship-blocker for any catalog shared with DuckDB writers.

**Short-term fix (this task)**: add a `format = 'puffin'` check in
`DucklakeSplitManager` / `DucklakePageSourceProvider` that throws a clear
`TrinoException` with a message naming the culprit table and pointing at the
config. Defer the Roaring-bitmap reader for later.

**Acceptance**:
- Test that constructs (or simulates) a `ducklake_delete_file` row with
  `format = 'puffin'` and asserts the connector throws a clear error rather than
  returning rows.
- No silent-success path remains.

**Pointer**: source item in [TODO-READ-MODE.md § Puffin Deletion Vector Reads](TODO-READ-MODE.md#puffin-deletion-vector-reads).
Reuse analysis for the long-term reader is in
[DUCKLAKE_1_0_IMPACT.md § Existing Puffin Support in Trino](DUCKLAKE_1_0_IMPACT.md#existing-puffin-support-in-trino-reuse-analysis).

## #2 — `uuid` scalar + `list<uuid>` inlined reads (read side)

**Why this is #2**: concrete cross-engine read failure (DuckDB writes a UUID,
Trino reads inlined → throws `"Expected entry size to be exactly 16"`). UUIDs
are a common primary-key choice, so this is not a corner case. Bounded fix in
one converter file.

**Fix**: in `DucklakeInlinedValueConverter`, add a `UuidType` branch that
parses the 36-char string with `java.util.UUID.fromString` and packs the 16
bytes via `io.trino.spi.type.UuidType.javaUuidToTrinoUuid`. The list path
inherits the same gap and is fixed by the scalar branch (the list parser
delegates per-element).

**Acceptance**:
- Lift `@Disabled` from `TestDucklakeCrossEngineTypeAudit.testDuckdbListUuidReadsInTrino`
  and confirm both halves pass (the test runs both inlined and Parquet paths,
  so this also validates Trino's Parquet array reader against DuckDB's UUID
  layout for free).
- Add a scalar-UUID inlined-read test if one isn't already pinned by an existing
  fixture.

**Bonus** (ride-along, optional): the sibling `list<blob>` inlined path has the
same shape — `DucklakeInlinedValueConverter` strips quotes but doesn't decode
DuckDB's `\xNN` hex escapes, so `VarbinaryType` receives literal escape
characters. Compare against `ducklake_util.cpp::ToSQLString` BLOB branch for
exact escape semantics. Pinned by `@Disabled testDuckdbListBlobReadsInTrino`.
Skip if it bloats the change; do it as a follow-up otherwise.

**Pointer**: [TODO-READ-MODE.md § Inlined-Read Type Gaps](TODO-READ-MODE.md#inlined-read-type-gaps).

## #3 — Logical conflict checking in commit flow (write side)

**Why this is #3**: the next tier of correctness work, this time on writes.
Today's commit-time check is lineage-only — it catches snapshot-id contention
but lets through semantically incompatible interleavings (concurrent INSERT
vs. DROP COLUMN, INSERT against a concurrently-dropped table, etc.). The
lineage looks fine; the *meaning* of the change is broken.

**Why bundle with concurrency tests**: we currently have **zero
writer-vs-writer concurrency tests** ([TODO-WRITE-MODE.md § Concurrency Test Coverage](TODO-WRITE-MODE.md#concurrency-test-coverage)
sits immediately below the conflict-checking item by design). The same
engineering session should land both the fix and the test infrastructure that
exercises it.

**Fix**: add a `checkForLogicalConflicts(...)` pass in
`executeWriteTransaction`, after winning snapshot advancement but before
final commit. Define the conflict matrix across schema/table/view/data-file
operations and reject incompatible interleavings. Conflict errors should
include intervening-change summaries at the same quality as today's
retry-conflict text.

**Acceptance**:
- At least one concurrent-write test that hits a semantic conflict
  lineage-only checking would miss (e.g. T1 issues `ALTER TABLE DROP COLUMN`
  while T2 has an in-flight `INSERT` against that column; T2 must abort with
  a clear conflict error rather than committing).
- The two adjacent writer-vs-writer items in TODO-WRITE-MODE land at the
  same time (same harness).

**Pointer**: [TODO-WRITE-MODE.md § Logical Conflict Checking in Commit Flow](TODO-WRITE-MODE.md#logical-conflict-checking-in-commit-flow).

## Considered and deferred

These came up in triage and were deliberately *not* in the top 3:

- **`list<blob>` inlined reads** — correctness bug like UUID, but blobs in
  lists are rarer and the failure mode is silent garbage rather than a throw.
  Ride-along with #2 if cheap; otherwise queue.
- **Bucket partitioning, sorted writes, schema-evolution gaps (`SET TYPE`,
  nested struct field manipulation), commit context (`set_commit_message`),
  M8 maintenance procedures** — real feature backlog, not correctness bugs.
  Pick up after #1–#3.
- **Variant / Geometry / virtual columns** — degraded-but-honest is the
  documented stance. Ship later or never.
- **Sorted-table awareness on read, change feed (R6), cross-backend view
  tests (R7), `.slt` corpus adoption, view dialect transpilation** — research
  / optimization items. Not blocking anything correctness-relevant.
