# ducklake-catalog

JDBC-backed accessor for the DuckLake catalog metadata (the `ducklake_*` tables that
DuckDB's `ducklake` extension writes to a SQL catalog backend — Postgres in our case).

## jOOQ-generated schema bindings

Compile-time-typed bindings for every `ducklake_*` table live under [`generated/`](generated).
They are generated by the `jooqCodegenDucklake` Gradle task and checked into git.

Checking them in is deliberate:

- **Compile-time safety against schema changes.** When DuckLake's extension adds / renames
  / drops a column in a release, our code that reads or writes that column stops
  compiling until it's updated. No silent `SELECT *` drift.
- **Migrations show up in review.** A DuckLake version bump that mutates the catalog
  schema produces a visible diff in `generated/` alongside the `libs.versions.toml`
  bump — easy to spot the surface area the migration actually touches.
- **No runtime DB needed to build.** Downstream modules (and CI) compile against the
  checked-in bindings; only the regeneration step needs Docker.

### Regenerating

Whenever the DuckDB / DuckLake version bumps in `jvm/gradle/libs.versions.toml`
(or any time you suspect the catalog schema has drifted), regenerate:

```shell
./gradlew :ducklake-catalog:jooqCodegenDucklake
```

Docker (or Podman) must be running — the task spins up a Postgres testcontainer, uses
DuckDB's `ducklake` extension to create the catalog tables in it, then runs jOOQ
codegen against that live Postgres. The container is torn down at build end.

Review the resulting diff under `generated/` before committing; unexpected changes
often signal a DuckLake schema migration that the catalog code needs to adapt to.

### Where the wiring lives

- `build.gradle.kts` — applies `org.jooq.jooq-codegen-gradle`, configures the
  `ducklake` execution (Postgres dialect, `ducklake_.*` include filter, package
  `dev.brikk.ducklake.catalog.schema`, target `generated/`).
- `jvm/build-logic/.../DucklakePostgresService.kt` — Gradle `BuildService` that
  starts the Postgres container and bootstraps the DuckLake metadata schema.
- `jvm/jooq-custom-naming/` — `dev.brikk.jooq.codegen.JooqCustomNaming` strategy
  (class-name / package conventions for the generated types).
